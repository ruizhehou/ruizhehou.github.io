QUIC 协议深度解析

QUIC（Quick UDP Internet Connections）最初由 Google 在 2012 年设计，2021 年经 IETF 标准化为 RFC 9000，并成为 HTTP/3 的底层传输协议。它运行在 UDP 之上，将 TCP 的可靠传输、TLS 的安全加密、HTTP/2 的多路复用整合到一个协议栈里，从根本上解决了 TCP 时代积累的若干顽固问题。

为什么需要 QUIC

TCP 的历史包袱

TCP 设计于 1974 年，运行在操作系统内核中。这带来了一个根本性的问题：协议演进极其困难。修改 TCP 需要更新全球所有操作系统，而中间件（防火墙、NAT、负载均衡器）也会对 TCP 的行为做各种假设，任何改动都可能破坏现有网络设备的兼容性。

Google 统计发现，互联网上有大约 29% 的 TCP 连接无法成功建立，主要原因是中间件对新 TCP 选项的干扰。这意味着即使设计出更好的 TCP 扩展，也很难在真实网络中部署。

队头阻塞问题

HTTP/2 在应用层实现了多路复用，多个请求可以共用一个 TCP 连接。但 TCP 是字节流协议，它不知道上层有多个独立的数据流。当一个数据包丢失时，TCP 会阻塞整个连接等待重传，即使其他数据流的数据已经到达——这就是TCP 队头阻塞（Head-of-Line Blocking）。

在 1% 丢包率的网络下，HTTP/2 的性能甚至不如 HTTP/1.1（多个独立 TCP 连接），因为所有流都卡在同一个重传队列里。

握手延迟

建立一个 HTTPS 连接需要：TCP 三次握手（1.5 RTT）+ TLS 1.2 握手（2 RTT），合计 3.5 RTT 才能发出第一个请求。TLS 1.3 优化到了 1 RTT，但 TCP 握手的开销依然存在。

移动网络下一次 RTT 可能高达 100ms，3.5 RTT 意味着用户要等待 350ms 才能看到第一个字节。

QUIC 的核心设计

基于 UDP 的用户态实现

QUIC 运行在 UDP 之上，在用户态实现所有逻辑。这带来了两个关键优势：

  • 快速迭代：不需要修改内核，更新应用程序就能升级协议
  • 中间件穿透：UDP 包对防火墙和 NAT 更友好，中间件不会对 UDP 内容做假设

代价是需要在用户态重新实现拥塞控制、丢包恢复、流量控制等原本由内核 TCP 处理的功能，实现复杂度更高。

0-RTT 和 1-RTT 握手

QUIC 将传输层握手和 TLS 握手合并，首次连接只需 1 RTT 就能开始传输数据。

首次连接（1-RTT）：
Client                          Server
  |--- Initial (ClientHello) ---->|
  |<-- Initial (ServerHello) -----|
  |<-- Handshake (Certificate) ---|
  |<-- Handshake (Finished) ------|
  |--- Handshake (Finished) ----->|
  |--- 1-RTT Data (请求) -------->|   ← 第 1 个 RTT 结束即可发数据
  |<-- 1-RTT Data (响应) ---------|

更激进的是 0-RTT 恢复连接。客户端在第一次连接时会收到服务端的 Session Ticket，再次连接时直接携带 0-RTT 数据，无需等待任何握手响应：

恢复连接（0-RTT）：
Client                          Server
  |--- Initial + 0-RTT Data ----->|   ← 直接发请求，不等握手
  |<-- Initial + 1-RTT Data ------|

0-RTT 的代价是无法防御重放攻击，因此只适用于幂等请求（GET），不适用于有副作用的操作（POST）。

多路复用与流

QUIC 在协议层面原生支持多路复用。一个 QUIC 连接可以包含多个独立的流（Stream），每个流有自己的序号空间和流量控制。

关键区别在于：QUIC 的丢包重传只影响对应的流，不阻塞其他流。这从根本上解决了 TCP 的队头阻塞问题。

流的类型分为四种：

  • 双向流（Bidirectional Stream）：客户端和服务端都可以发送数据，用于 HTTP 请求/响应
  • 单向流（Unidirectional Stream）：只能由创建方发送，用于服务端推送、QPACK 编码等

流 ID 的低两位编码了流的方向和发起方：

0x0: 客户端发起的双向流
0x1: 服务端发起的双向流
0x2: 客户端发起的单向流
0x3: 服务端发起的单向流

连接迁移

TCP 连接由四元组（源 IP、源端口、目标 IP、目标端口）唯一标识。当手机从 WiFi 切换到 4G 时，IP 地址改变，TCP 连接必须重新建立。

QUIC 使用 Connection ID 标识连接，与网络地址无关。切换网络时，客户端只需用新的地址发送数据包，服务端通过 Connection ID 识别是同一个连接，无需重新握手。

为了防止连接追踪（隐私保护），QUIC 在迁移时会更换 Connection ID，使得网络路径上的观察者无法将迁移前后的流量关联起来。

内置加密

QUIC 强制要求 TLS 1.3 加密，没有明文模式。这不只是安全考虑，也是为了防止中间件干扰——加密的 QUIC 数据包让中间件无法解析内部结构，从而无法做出错误的假设或修改。

QUIC 的包头也做了部分加密（Header Protection），防止中间件根据包头信息做流量分析或干预。

数据包结构

包类型

QUIC 定义了两大类数据包：

长包头（Long Header）：用于握手阶段，包含完整的版本号和连接 ID。

  • Initial 包：握手的第一个包，携带 ClientHello/ServerHello
  • Handshake 包：握手后续阶段，携带证书和 Finished 消息
  • 0-RTT 包：0-RTT 数据，使用会话恢复密钥加密
  • Retry 包：服务端要求客户端重试（地址验证）

短包头（Short Header）：握手完成后的数据传输阶段，去掉版本号等冗余字段，减少开销。

帧结构

QUIC 数据包的载荷由一个或多个帧（Frame）组成，不同类型的帧承载不同功能：

  • STREAM 帧：携带应用数据，包含流 ID、偏移量、数据长度和数据内容
  • ACK 帧：确认收到的数据包，支持选择性确认（SACK），包含多个 ACK Range
  • CRYPTO 帧：握手阶段的 TLS 数据
  • CONNECTION_CLOSE 帧：关闭连接，携带错误码和原因
  • MAX_DATA 帧：更新连接级别的流量控制窗口
  • MAX_STREAM_DATA 帧：更新流级别的流量控制窗口
  • PATH_CHALLENGE / PATH_RESPONSE 帧：连接迁移时的路径验证
  • PADDING 帧：填充数据包到指定大小，防止流量分析

可靠性机制

包号与确认

QUIC 使用单调递增的包号（Packet Number）标识每个数据包，重传的数据包使用新的包号。这解决了 TCP 的重传歧义问题——TCP 重传包与原始包使用相同序号，导致 ACK 时无法区分确认的是哪次发送。

QUIC 的 ACK 帧支持多个 ACK Range，可以精确描述哪些包已收到、哪些包丢失，相当于 TCP 的 SACK 但更灵活。

ACK 延迟（ACK Delay）字段记录了接收方从收到数据包到发出 ACK 的时间，发送方可以用这个信息更精确地计算 RTT。

丢包检测

QUIC 使用两种机制检测丢包：

基于包号的丢包检测：如果一个包号之后已经有足够多的包被确认（默认阈值为 3），则认为该包丢失，触发重传。这类似于 TCP 的快速重传。

基于时间的丢包检测：如果一个包超过 max(9ms, 1.125 × smoothed_RTT) 没有被确认，则认为丢失。这是 QUIC 的 PTO（Probe Timeout）机制，比 TCP 的 RTO 更激进，减少了尾部延迟。

拥塞控制

QUIC 默认使用 New Reno 拥塞控制算法，但协议本身不绑定具体算法，可以在用户态灵活替换。Google 的 QUIC 实现使用 BBR（Bottleneck Bandwidth and Round-trip propagation time），专为高带宽、高延迟网络优化。

与 TCP 不同，QUIC 的拥塞控制以字节数而非包数计算窗口，更精确地反映实际的带宽占用。

QUIC 还引入了显式拥塞通知（ECN）支持，路由器可以在不丢包的情况下通知发送方降速，减少因丢包触发的重传。

流量控制

QUIC 有两层流量控制：

流级别（Stream-level）：每个流独立的接收缓冲区大小。发送方不能超过接收方通告的 MAX_STREAM_DATA 值。

连接级别（Connection-level）：所有流的总数据量上限。防止单个连接占用过多内存。发送方不能超过 MAX_DATA 值。

两层控制相互独立，都不能超过才能发送数据：

可发送数据量 = min(
  MAX_STREAM_DATA[stream_id] - 已发送字节数,
  MAX_DATA - 所有流已发送字节数之和
)

当接收缓冲区被消费后，接收方发送新的 MAX_DATA / MAX_STREAM_DATA 帧更新窗口。如果发送方等待窗口更新超时，会发送 STREAM_DATA_BLOCKED 帧通知接收方。

HTTP/3 与 QUIC

HTTP/3 是运行在 QUIC 之上的应用层协议，取代了 HTTP/2 的 TCP+TLS 组合。

HTTP/3 的核心改动：

  • 去掉了 HTTP/2 的帧层：QUIC 本身提供了流和帧的抽象，HTTP/3 直接复用
  • QPACK 替代 HPACK：HTTP/2 的 HPACK 头部压缩依赖单一的有序流，在多路复用下会引入队头阻塞。QPACK 将编码表的更新和实际请求解耦，通过独立的单向流传输，彻底消除头部压缩的队头阻塞
  • 服务端推送保留但受限：HTTP/3 保留了 Server Push，但实践中很少使用

HTTP/3 的流分配：

Stream 0, 4, 8, ...  : 客户端发起的双向流（HTTP 请求/响应）
Stream 2             : 客户端控制流（SETTINGS 等）
Stream 3             : 服务端控制流
Stream 6             : QPACK 编码器流（客户端 → 服务端）
Stream 7             : QPACK 解码器流（服务端 → 客户端）

性能对比

Google 的实测数据（YouTube）：

  • 视频缓冲时间减少 15%~18%
  • 视频重新缓冲率减少 18%
  • 弱网（2% 丢包率）下延迟降低 50%

Facebook 的数据：

  • 图片加载时间减少 8%
  • 视频启动时间减少 20%

QUIC 在以下场景收益最大：

  • 高丢包网络：移动网络、卫星网络，QUIC 的流级别隔离避免了全局阻塞
  • 高延迟网络：0-RTT 和 1-RTT 握手显著减少连接建立时间
  • 频繁切换网络：连接迁移避免重新握手
  • 短连接场景：0-RTT 对短连接的收益尤为明显

在低延迟、低丢包的数据中心内网，QUIC 相比 TCP 的优势不明显，且 UDP 在某些内核版本下的处理效率低于 TCP，可能有性能回退。

部署现状

截至 2026 年，QUIC 已被广泛部署：

  • Google：所有 Google 服务（搜索、YouTube、Gmail）默认使用 QUIC
  • Meta：Facebook、Instagram 大量使用
  • Cloudflare：全平台支持 HTTP/3
  • 浏览器：Chrome、Firefox、Safari 全部支持 HTTP/3
  • 服务端：Nginx（实验性）、Caddy、LiteSpeed、H2O 均有支持

根据 W3Techs 的统计，全球约 30% 的网站支持 HTTP/3。

实现挑战

QUIC 的实现比 TCP 复杂得多，主要挑战：

CPU 开销：UDP 的 GSO（Generic Segmentation Offload）和 GRO（Generic Receive Offload）支持不如 TCP 成熟，加密解密在用户态进行，CPU 消耗较高。Google 报告 QUIC 服务端的 CPU 消耗是 TCP 的 2 倍。

缓冲区膨胀：操作系统的 UDP socket 缓冲区默认较小，高吞吐场景需要手动调大。

NAT 超时：部分 NAT 设备对 UDP 的超时时间比 TCP 短（30 秒 vs 2 分钟），长连接需要发送 PING 帧保活。

防火墙拦截：部分企业防火墙默认封锁 UDP 443 端口，QUIC 连接失败后需要回退到 TCP。浏览器实现了 Happy Eyeballs 机制，并行尝试 QUIC 和 TCP，哪个先成功用哪个。

总结

QUIC 是传输层协议的一次重大革新，它通过在 UDP 上重新实现可靠传输，绕开了 TCP 的历史包袱，解决了队头阻塞、握手延迟、连接迁移三个核心问题。

从架构上看，QUIC 的设计哲学是把协议逻辑从内核移到用户态，以牺牲一定的 CPU 效率换取快速迭代和灵活部署的能力。这个权衡在 Google 规模的场景下已经被证明是值得的。

对于应用开发者，QUIC 大多是透明的——HTTP/3 的 API 与 HTTP/2 完全兼容，启用 HTTP/3 通常只需要在服务端配置中打开一个开关。理解 QUIC 的价值在于知道什么时候它能带来收益、什么时候它不适用，以及出现问题时如何排查。
